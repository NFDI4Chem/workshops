Recommendations Standard for Reporting Liquid-State NMR Experiments of Small Molecules​

This recommendations standard is a proposal from NFDI4Chem of what metadata to report for liquid-state NMR experiments of small molecules. We aim to make it suitable for NMR repositories and the supplementary materials of publications, not for the metadata in the experiment section. Therefore, we want to emphasize that the chemist doesn't need to enter their data in the recommended format as long as the repository is capable of converting it. For instance, the user might only upload NMR datasets, then NMRium extracts the solvent, and the repository does the mapping to the CHEBI instance.

While forging this standard, we have considered two aspects: How to make sense of the data on the first hand, where we need the minimum mentioned here in the first level, and data traceability and reproducibility, where details like the number of scans and the instrument become very useful, so we asked for them in the second level.

The recommendations started with metadata for reporting NMR analyses of synthetic compounds only, covering the reaction leading to the analyzed product. Then the decision was taken to split the recommendations and remove the synthetic compounds section to make them more modular and broader to cover all small molecules, including natural products, synthetic compounds, and mixtures where we have known molecules. They also started by making crucial and automatically extractable metadata (by NMRium) mandatory, which can encourage chemists to use automatic workflows, and we made metadata relevant to impurities recommended. However, for historical data, authors might fail to submit with restricted minimum information (MI), which led to the decision to keep automatically extractable metadata recommended if it wasn't crucial.

Here is the link to the recommendations draft, which will be open for editing until publication. It will be further discussed with IUPAC to reach a joint agreement.

Scientific Applications Development - NMRium​

In Zakodium, we are trying to make programming FAIR. We use conventional committing to automatically generate the change log based on bug fixes and new features and to automatically close issues. We have a lot of continuous integration and automatic GitHub testing. GitHub coverage tells you how much of the code was tested, and it is usually over 80% of our code. We also know which lines haven't been tested. We use the GitHub project to have an overview of the works where we can archive what was closed. We publish our projects on Zenodo to get DOIs.

Building react application: We use many little libraries for GUI, react components, etc., and we use libraries we have control on. We have created popular libraries, including parsers, JCAMP or Bruker converters, or ones creating jcamp from other data. In NMRium, we use more than 60 of our libraries. Pixelium is another program that works with image processing, not related to NMR, but uses a lot of libraries we created for NMRium.

NMRium has an MIT license, and it provides automatic bug reports to tell what went wrong. We have several components in NMRium like converters, structure viewing, load and save data, etc. nmr-load-save library can extract most of the metadata in NMRium from the raw files. NMRium then uses one JSON containing all the data related to the original and processed data. Some NMRium features: multiplet analysis, you can display multiplicity trees. You can show the structure and do the assignment there. NMRium is interactive between the tables, the molecule, and the spectrum. It can generate reports. It also allows you to have a database, like a solvent or reference database, which you can search by structure, substructure, or shifts, or with advanced search to look for specific atoms.

Prediction: It is predicting the spectrum by providing only the structure. It is also interactive. We predict 1H, 13C, COSY, HSQC, HMBC. There is a new feature from yesterday about simulation. You enter values in the table to get a spectrum which is helpful for teaching. We provide e-learning where you process the spectrum and enter your suggested structure then you find if correct or not. Metabolomics: Even in the browser, we are still able to handle huge gigabytes as NMRium can filter those huge files, for instance, based on the type.

Towards Open NMR Data Formats - What We Want and What We Can Have​

In NMR, we prepare a sample with a molecule that we analyze in the spectrometer to get information that we want to assign to the molecule. NMR data has several layers: raw, processed, and assignment. And another layer is the metadata. In higher levels of NMR, it is difficult to tell what is data or metadata.

Available NMR formats: JCAMP is a text-based file with the actual data and the binary data included, which will be lengthy if not compressed, with tags to define the metadata. It is open and Common among chemists and popular in ELNs. Much data is already in JCAMP, which makes it difficult to run away from or replace by Bruker! However, it is not maintained. There are many vendor-dependent variants, the 2D NMR specification is an incomplete draft, and there are no coordinated updates of the variants. There are incompatibilities between different software packages, and it has no (semantic) validation.

Bruker has a folder structure with different files. FID has the binary data from the acquisition. Acqu is related to JCAMP and provides metadata too. The Bruker dataset is all you need to reproduce an NMR experiment.

nmrML has embedded ontology. It is probably the only format in the NMR world with ontology use. It is not famous out of metabolomics and biology world as it is biology-oriented, but this orientation can get fixed.

NMReData doesn’t deal with spectral data. It alternatively extends the sdf structure and the sdf with 1D, 2D, assignments, coupling constants, correlations, etc.

The landscape:

  • JCAMP and Bruker cover raw and processed data and, if at all, assignment data.
  • nmrML covers a little assignment but is not designed for that.
  • NMReDara covers only assignment data.

Discussion:

What do we want from an open NMR format? Do we need one to start with? If Bruker becomes very well documented and as their specifications are already open, could it be enough? If everyone in NMR can work with Bruker, then why have an open one?

It will take too much human power. It is better to modernize Bruker than maybe work with nmrML and other formats. Documenting an existing format is easier and cheaper than developing a new one.

Several analytical techniques using JCAMP are suffering from overlapping documentation. JCAMP is very old for parsers for being textual, making it not encouraging to modernize it. There is no way to validate JCAMP other than writing parsers which need implementing all its "standard" versions. It is not recommended to use ontologies in JCAMP, because this modifies JCAMP in a way the chemists might not be interested in or understand. JCAMP missing important things can be an argument to enhance the existing format to get better data.

To enhance JCAMP, we should decide the minimum to include in that format. We can use the dot for optional fields to add what interests us in NMR. What about taking the JCAMP model and providing another JSON serialization of it? But we need validation and testing for the industry.

Can we work with IUPAC by taking an available format, working on it, adding from Bruker, and asking for IUPAC's recommendation? We all agreed we do not want to develop a new format but to enhance existing ones. We should modify it as little as possible to get what we need. Another option is the containers, where we provide the raw NMR data packaged with other open formats and JSON metadata (NMRium, schema.org, or others)

nmrXiv + NMRium as The Open Source Choice for NMR Platforms​

In the NMR lab, workflows are not standardized. Some use LIMS, others servers. With LIMS, data submission becomes easy without paper submission. Another option is providing the data to the user with accounts. The repository is an important concept because if it includes apps like NMRium, it helps not only in data visualization but it in processing and assignment.

There are 2 aspects we need to consider, the side of the administrators where quality checks and ranking can happen. And the aspect of user expectations, if you give 50 checkboxes to chemists, we should expect low submission. It is also common not to provide 2D assigned spectra, which are needed in the common routines. A repository or a tool like NMRium can provide added value by suggesting assignments.

What is an ideal laboratory NMR workflow to generate data ready for submission:

  • Data hosting instead of chemists-preferable local storage.
  • You also need at least one structure linked to the spectrum. It is still common that people don’t submit structures.
  • The local and public format for data storage should be the same to avoid switching between systems.
  • What to do with the data once created? People usually process on specific processing software, and it would be great for repositories to provide tools for directly visualizing and processing data on the same platform. NMRium in nmrXiv gives living data.
  • Good to provide one interface for all tasks. If the platform can give suggested assignments, this would be very attractive to users.
  • There should be some assignment control to check the plausibility of chemical shift comparisons but also can be used with case environment including logic, and if people do some assignment, the system should be capable of continuing the job.

Success factors:

  • Chemists don’t like to handle several interfaces; instead, they might go to paper. It will be good to have a streamline for all the steps.
  • Very important to provide the structure always in the repo.
  • The privacy issue is important. Giving people the option to publish data with a restricted group instead of just missing the data is needed.

Conclusion: Ideal Workflow:

  • Sample: Electronic submission, structure, roles
  • Data: Transfer, local repository/LIMS
  • Processing: Visualization, processing on the fly
  • Assignment: Proposal, validation, export
  • Repository: Local, review, public

Discussion: We need to include ELNs in the vision. Core facilities should have the role of raising the bar in making data digital, despite not all of them being allowed to do so, but they still should push to get better. DFG hands out money only if you adhere to a certain level. So the control over the data quality can happen from the funders' side. In the Jena platform, there is a commercial management system, and it is used instead of paper, but one issue with there is that you can search by people and group, not by molecules, for structure submission is not mandatory.

In nmrXiv, the repo layer is ready, but we are enhancing the submission and the molecule layers. We also want to build the prediction and search layer separately from nmrXiv to allow others to use the same projects. Collaboration is still needed with Nils to push things faster. We already work with Chemotion, an open-source ELN, which we found to be the best in chemistry field.

NMR Data Packaging with Metadata in NFDI4Chem​

NMR metadata is already provided by NFDI4Chem services such as nmrXiv (NMR data repository) and Chemotion (Repository for samples, reactions, and the accompanying analytical data) in a structured way in JSON format complying with schema.org and Bioschemas types. The use of NMRium in nmrXiv and (soon) in Chemotion ensures a high extract of MI that can get translated to schema JSON and can be further optimized. Recommended and optional data will depend on the user, but we already provide the means to input them in Chemotion templates and nmrXiv fields.

However, metadata can get lost during data download. When the user downloads the data, no structured metadata files are added automatically within the folder. In Chemotion, even the datasets belonging to one reaction have to get downloaded separately. If the user wants to obtain the metadata, they have to go through a separate download/export process (separate download buttons, API calls..) It is common in Chemotion to find more added chemical materials in the “description” field than what the reaction table provides, which needs to get considered when deciding what metadata to export. In Chemotion, we have datasets with each having its own NMR files download buttons and own schema.org JSON button, with the download of the NMR files and their schema being 2 separate processes. In nmrXiv, we have one Download button for the whole project, where all the NMR data files get downloaded, but the metadata files are not included. Additionally, there are no buttons to download the metadata, but only through API calls with which the chemists are not familiar.

We need to package related data and metadata together in one folder.

  • Data packaging tools request metadata to be structured making it machine readable.
  • Some allow data integrity checks to monitor the data after undergoing different processes (download, zip/unzip, transfer..)
  • And they also provide the data in a convenient format for these processes such as transfer and download, which are the main focus in NFDI4Chem at the moment.

There are several data packaging approaches we have considered for NFDI4Chem data. BagIt provides file layout conventions for data packaging. A "bag" is the root folder. It has two types of files:

  • “Payload” files: The main data files.
  • Tag files: descriptive metadata about the payload data files.

The bag is the root folder which consists of the following

  • Bag Declaration: bagit.txt: The "bagit.txt" tag file MUST consist of exactly two lines in this order:
    • BagIt-Version: M.N
    • Tag-File-Character-Encoding: ENCODING
  • Payload Manifest: manifest-algorithm.txt: A payload manifest file provides a complete listing of each payload file name along with a corresponding checksum to permit data integrity checking. A bag can have more than one payload manifest, with each using a different checksum algorithm.
  • A bag MAY contain other tag files, so we can add DataCite metadata or RO-Crate JSON.
  • The base directory MUST contain a subdirectory named "data", where we can provide our NMR data.

BagIt has its format to integrate checksum algorithms allowing data integrity checking. It additionally offers flexibility in the folder structure to add different metadata files as it suits the user. However, it doesn’t provide a structured way NMR related metadata. And even its structured specifications are txt based (https://datatracker.ietf.org/doc/html/rfc8493#section-2.2.2 )

RO-Crate (Research Object Crate) is a method of aggregating and describing research data with associated metadata. For example: who created it, using which equipment, and under which license it can be re-used. RO-Crate uses JSON-LD to express this metadata, describing data resources as well as contextual entities such as people, organizations, software, and equipment as a series of linked JSON-LD objects - using commonly published vocabularies, chiefly schema.org

A root directory is identifiable by the presence of the RO-Crate Metadata File, ro-crate-metadata.json, a JSON-LD file. This file contains structured metadata about the dataset as a whole (the Root Data Entity) and, optionally, about some or all of its files, which provides a simple way to, for example, assert the authors (e.g., people, organizations) of the RO-Crate or one of its files, or to capture more complex provenance for files, such as how they were created using software and equipment. Payload files and directories: These are the actual files and directories that make up the dataset being described.

The issue with RO-Crate is that all folders within the root will be identified as "Datasets", while nmrXiv from NFDI4Chem refers only to a complete NMR assay (a Bruker folder or JCAMP file) as a dataset. This will cause a huge confusion when, for instance, you have a "Bruker" folder, and thus it is a dataset according to RO-Crate, and then you have a pdata folder within, and it is also a dataset, while nmrXiv considers the whole assay folder as a dataset.

nmrCV Controlled Vocabulary​

nmrCV was a part of nmrML development, and we still can use it as a controlled vocabulary. nmrCV development was done before 2019, then there was a group that worked on pulse sequences, and there is a TIB team who works on ontologies in NFDI4Chem.

nmrCV needs:

  • Structure cleaning in terms of hierarchy.
  • There are missing terms here and there.
  • There are white spots there that we can maintain and import
  • Misuse of instances and classes.
  • IRIs nor resolvable (not a must, but a good to have.)

Suggested: NFDI4Chem maintenance of nmrCV, with nmrXiv contribution.
